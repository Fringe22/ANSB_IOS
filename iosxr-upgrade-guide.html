−Overview & Safety Model

This playbook upgrades Cisco ASR9K routers from IOS XR 7.8.2.x to 24.4.2.1, one device at a time (configurable to parallel), with full rollback capability until the final install commit.

SINGLE RSP5 — FULL RELOAD REQUIRED

  • ISSU NOT SUPPORTED — single RSP means no graceful switchover
  • TRAFFIC IMPACT — all upgrades require a maintenance window
  • RELOAD TIME — 15–20 minutes per reload (install activate, FPD, SMU)

Safety Guarantees

  • Rollback-safe: Everything before install commit (phase 13) is reversible
  • MD5 mandatory: Every image file and every SMU is hash-verified on-device before install
  • Traffic isolation: Node is drained (IS-IS overload + Loopback1 shutdown) before any package work
  • Neighbor set verification: Every specific baseline IS-IS adjacency and EVPN BGP peer must return after un-isolation
  • Automatic rollback: block/rescue wraps all destructive phases — failure triggers install rollback to committed, un-isolates, and reports
  • Soak period: 15-minute configurable window between POST checks and commit

Workflow Diagram

PRE-CHECKS → DISK-PREP → FPD-PREP → ISOLATE | +----- BLOCK (auto-rollback on failure) -----+ | | | SMU-BEFORE → INSTALL → FPD → SMU-AFTER | | | | | v | | UN-ISOLATE → POST-CHECKS → SOAK | | | +---------------------------------------------+ | failure? → RESCUE (rollback + un-isolate) | COMMIT

−Traffic Isolation (ISOLATE / UN-ISOLATE)

Before any package work, the node is gracefully drained so neighbors route around it. This prevents hard forwarding failure during reload.

ISOLATE (before package work)

1Shut Loopback1 — tears down EVI / iBGP sessions sourced from Lo1

interface Loopback1
 shutdown
 commit comment "Maintenance-window isolation: shut Lo1"

2Set IS-IS overload — neighbors route around this node

router isis main
 set-overload-bit
 commit comment "Maintenance-window isolation: IS-IS overload"

3Wait for BGP drain — poll show bgp sessions until no Established sessions (up to 5 min)

4Confirm IS-IS overload advertised — show isis | include Overload

Isolation persists across reloads. The shutdown and set-overload-bit are committed to running-config, so they survive SMU-activate and install-activate reloads inside the maintenance window. UN-ISOLATE explicitly reverses them.

UN-ISOLATE (after all package work)

1Clear IS-IS overload (first, so IS-IS reconverges before BGP)

router isis main
 no set-overload-bit

2No-shut Loopback1

interface Loopback1
 no shutdown

3Wait for ALL baseline IS-IS adjacencies to return (set-based, not count-based)

# Polls until: (baseline_neighbors - current_neighbors) == empty set
# Retries: 40 x 15s = 10 minutes max

4Wait for ALL baseline EVPN BGP peers to re-establish

# Polls until: (baseline_peers - current_peers) == empty set
# Retries: 80 x 15s = 20 minutes max
Set-based, not count-based. The playbook captures the exact set of IS-IS neighbors and BGP peers at baseline, then asserts every specific one returns. If you lose neighbor A and gain neighbor B, counts match but ANSB_IOS catches this.

−MD5 Verification

Every file — image files (TAR or individual RPMs) and SMUs — must have a pre-supplied MD5 hash verified on-device before any install operation.

How It Works

  1. Operator provides filename: md5hash pairs in the AWX survey (or group_vars)
  2. Playbook asserts every listed file has a valid 32-char hex MD5
  3. On-device: show md5 file /harddisk:/<filename>
  4. Assert computed MD5 matches expected — abort if mismatch

Survey Format

Image files and SMUs both use the same combined format (one line per file):

# Image files (single TAR)
asr9k-x64-24.4.2.1.tar: 7174dd5a5cf7952209b67602e038af0d

# Or individual RPMs
asr9k-mini-x64-24.4.2.iso: 7174dd5a5cf7952209b67602e038af0d
asr9k-isis-x64-2.4.2.1-r2442.x86_64.rpm: 3fa8b2c1e9a04d7f8b6c2e1a5d9f3b7c

# SMUs BEFORE
asr9k-sysadmin-asr9k-7.8.2.4-r782.CSCwh61644.x86_64.rpm: 6f58dd1f3d1b2df3f1c6c447b2a6f31b
asr9k-sysadmin-install-7.8.2.2-r782.CSCwi95358.x86_64.rpm: 6cb3873222ec7803ab7e2107551f3f5c

IOS XR MD5 Command

RP/0/RSP0/CPU0:router# show md5 file /harddisk:/asr9k-x64-24.4.2.1.tar
7174dd5a5cf7952209b67602e038af0d
Generate MD5s on the file server before staging: 
md5sum asr9k-x64-24.4.2.1.tar
md5sum asr9k-sysadmin-*.rpm

−Install Workflow (synchronous + operation ID)

The playbook uses synchronous keyword and operation-ID tracking — no polling loops needed.

Step-by-Step

1install add — extracts packages from TAR/ISO on device (pre-staged on harddisk:)

install add source harddisk: asr9k-x64-24.4.2.1.tar synchronous
# Returns: "Install operation 42 started..."

Key difference from other approaches: images are pre-staged on harddisk:, not pulled from TFTP during the window. No dependency on external file servers.

2Extract operation ID — parsed from install add output

# Regex: Operation[ :]+([0-9]+) → "42"
# Used for prepare and activate to reference the exact operation

3install prepare — validates packages, non-disruptive

install prepare id 42 synchronous
# Blocks until validation complete. Fails on incompatibility.

4install activate — triggers reload

install activate id 42 synchronous noprompt
# SSH session drops. Device reloads.

5Wait for reload — two-phase SSH monitoring

# Phase A: wait_for port 22 state=stopped  (confirm SSH went DOWN)
# Phase B: wait_for port 22 state=started  (confirm SSH came BACK)
# Then: 180s settle pause for processes to initialize
Both phases matter. Without checking SSH went down first, the playbook could proceed before the reload started (SSH still up from old session).

6install commit (phase 13, after POST + soak)

install commit
# Makes upgrade permanent. Point of no return.

Why synchronous?

  • CLI blocks until the operation completes — no polling show install request in a loop
  • Eliminates race conditions between poll intervals
  • Ansible task finishes when the IOS XR operation finishes
  • noprompt suppresses the "Continue?" confirmation

Why operation ID (not name)?

  • install prepare id 42 is unambiguous — references exactly the operation that just ran
  • Name-based (install prepare asr9k-mini-x64-24.4.2) can match wrong version if multiple are in repository

−SMU Patching

SMUs are applied in two windows: BEFORE the version upgrade (committed immediately) and AFTER (committed atomically with the version).

SMU-BEFORE (phase 6)

  • Patches the current image (e.g., 7.8.2.x)
  • Committed immediately so they survive a version rollback
  • Example: CSCwh61644, CSCwi95358 (sysadmin/install fixes)

SMU-AFTER (phase 9)

  • Applied on top of the new image (24.4.2.1)
  • NOT committed — bundled with install commit at phase 13
  • If SMU-after fails, nothing is committed — full rollback is safe

Per-SMU Processing (smu_phase.yml)

  1. MD5 verify each SMU file on device
  2. install add source harddisk: smu1.rpm smu2.rpm smu3.rpm synchronous (batch)
  3. Extract operation ID
  4. Detect restart type: show install package <smu> detail → "Restart Type: Reload" or "Restart Type: Restart"
  5. install prepare id <N> synchronous
  6. install activate id <N> synchronous noprompt
  7. If reload: wait for SSH down/up. If process-restart: activate blocks until done.
  8. Verify each SMU in active set
  9. Commit (SMU-BEFORE only)
Batch add, single activate. All SMUs in a phase are added in one install add command, then prepared and activated together. Faster and handles inter-dependencies correctly.

−FPD Management

Field-Programmable Device upgrades run after the image install, while the node is still isolated.

FPD-PREP (phase 4)

Disables automatic FPD behavior before the image install:

fpd auto-upgrade disable
fpd auto-reload disable

Applied in both XR and sysadmin config planes. Verified via show running-config fpd.

FPD Upgrade (phase 8)

Stage 1: Sequenced upgrades (field-notice-driven)

For FN74306 on 4HG-FLEX, each FPD must be upgraded and verified CURRENT before the next starts:

# Example sequence (slot 0/1):
1. admin upgrade hw-module location 0/1 fpd Sunstreaker force
   → poll until CURRENT
2. admin upgrade hw-module location 0/1 fpd TAMFW-Sunstreaker force
   → poll until CURRENT
3. admin upgrade hw-module location 0/1 fpd Primary-BIOS force
   → poll until CURRENT

Uses fpd_sequenced_single.yml with include_tasks loop — upgrade + poll per FPD.

Stage 2: Bulk upgrade

admin upgrade hw-module location all fpd all

Stage 3: Admin reload

admin hw-module location all reload
# SSH drops, wait for reload (same two-phase SSH monitoring)

Stage 4: Verify

show hw-module fpd
# Assert: no "NEED UPGD" or "NEED REL" in output

−Rollback (Automatic + Manual)

Automatic Rollback (block/rescue)

Phases 6–12 are wrapped in block/rescue. If any task fails:

  1. Record failure details (task name + error message)
  2. Report failure to job output
  3. install rollback to committed noprompt
  4. Wait for SSH down/up (rollback triggers reload)
  5. 180s settle pause
  6. Un-isolate: clear IS-IS overload + no-shut Loopback1
  7. Verify rollback version
  8. Report rollback summary
  9. Fail the play (stops the batch via any_errors_fatal)
After automatic rollback: the node is back in service on the old image, un-isolated, with neighbors recovering. The operator gets a clear summary of what failed and what to check.

Manual Rollback (before install commit)

admin install rollback to committed

Device reloads to the previously-committed image. Then manually un-isolate:

configure
 router isis main
  no set-overload-bit
 interface Loopback1
  no shutdown
commit
end

Manual Rollback (after install commit)

install rollback to <old-label>     # e.g. 7.8.2
install commit

Same maintenance-window rules apply — isolate first, verify neighbors return.

Assertions (gates — abort if failed)

  • Not already on target version
  • All image files exist on device
  • Platform healthy (IOS XR RUN, no Not Ready)
  • NTP synchronized
  • No pending install operations (show install request)
  • No critical/major alarms
  • At least one IS-IS neighbor + one EVPN BGP peer at baseline

Snapshot Saved

Full output of all 13 commands + alarms saved to ./backups/<hostname>_pre_<timestamp>.txt on the AWX worker for post-hoc diffing.

−AWX / AAP Setup

Automated (recommended)

provision_tower.yml creates everything in one run:

# Prerequisites
pip install awxkit
ansible-galaxy collection install -r collections/requirements.yml

# Connection
export CONTROLLER_HOST=https://awx.example.com
export CONTROLLER_USERNAME=admin
export CONTROLLER_PASSWORD='...'

# Run
ansible-playbook provision_tower.yml \
    -e tower_project_scm_url=https://github.com/YOUR_ORG/ANSB_IOS.git

What It Creates

  • Project — git-backed, auto-syncs on launch
  • Inventory — hosts + iosxr_devices group with connection vars
  • Machine Credential — IOS XR admin user/password
  • Job Template — IOS XR Upgrade with 14-field survey, Limit prompt on launch

Launch

  1. Templates → IOS XR Upgrade → Launch
  2. Fill Limit: pe1.lab (or multiple for parallel)
  3. Fill survey (most have defaults)
  4. Launch — watch job output live in AWX
